# Lezione 05 — Pipe e Redirection

## Problema

Sappiamo usare i comandi core:
- creare file
- leggerli
- cercare informazioni

Ma ogni comando lavora **da solo**.

Problema concreto:
- l’output di un comando è troppo lungo
- vogliamo filtrare risultati
- vogliamo salvare l’output in un file
- vogliamo gestire errori separatamente

Contesto reale:
- analisi di log
- ispezione di processi
- automazione
- scripting

Vincoli noti:
- i comandi producono testo
- la CLI lavora per stream
- copiare/incollare manualmente non scala

Cosa **non** sappiamo ancora fare:
- collegare comandi tra loro
- reindirizzare output e input
- separare output normale ed errori

Perché il problema è rilevante:
> Senza pipe e redirection, la CLI è solo una sequenza di comandi isolati.

---

## Teoria

Spiegazione dei concetti necessari **solo** per collegare e controllare i flussi di dati.

---

### Stream standard

Ogni processo CLI utilizza tre stream standard:

- **stdin (0)** → input
- **stdout (1)** → output normale
- **stderr (2)** → output di errore

Diagramma concettuale:

```

stdin → [ comando ] → stdout
↘→ stderr

```

Questi stream esistono **sempre**, anche se non li vedi.

---

### Redirection

La **redirection** permette di cambiare destinazione degli stream.

Operatori principali:

- `>`  → scrive stdout (sovrascrive)
- `>>` → scrive stdout (append)
- `<`  → fornisce input
- `2>` → redirige stderr

La redirection **non modifica il comando**, solo i flussi.

---

### Pipe

La **pipe (`|`)** collega:
- stdout di un comando
- a stdin di un altro comando

Modello mentale:

```

comando1 | comando2 | comando3

````

Ogni comando fa una cosa sola,  
la pipe costruisce il comportamento complesso.

---

### Regola fondamentale

> Le pipe collegano **processi**, non file.

I file sono solo:
- sorgenti di input
- destinazioni di output

---

## Esempi

Applicazione pratica immediata dei concetti teorici.

---

### Redirigere output in un file

```bash
ls > files.txt
````

Sovrascrive il file se esiste.

Append:

```bash
ls >> files.txt
```

---

### Usare un file come input

```bash
sort < files.txt
```

Il comando legge da stdin invece che da tastiera.

---

### Gestire gli errori

```bash
ls file_inesistente 2> error.log
```

Stdout e stderr sono separati.

Entrambi nello stesso file:

```bash
ls file_inesistente > all.log 2>&1
```

---

### Pipe semplice

```bash
ls -l | less
```

* `ls` produce output
* `less` lo consuma

---

### Pipe con filtro

```bash
ls -l | grep ".md"
```

Filtra solo i file Markdown.

---

### Pipe multiple

```bash
ps aux | grep python | awk '{print $2}'
```

Pipeline:

1. `ps` produce dati
2. `grep` filtra
3. `awk` estrae colonne

---

### Caso realistico: log

```bash
cat syslog | grep error | less
```

Forma più corretta:

```bash
grep error syslog | less
```

Regola pratica:

> evita comandi inutili nella pipeline.

---

## Output atteso della lezione

Alla fine di questa lezione sai:

* cosa sono stdin, stdout e stderr
* redirigere input e output
* separare errori e risultati
* costruire pipeline di comandi
* leggere la CLI come flusso di dati

Da qui in poi, la CLI diventa
**componibile, potente e scalabile**.
