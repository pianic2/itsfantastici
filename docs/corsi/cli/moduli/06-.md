# Lezione 06 — Variabili e ambiente

## Problema

Ora sappiamo:
- usare comandi core
- collegarli con pipe
- controllare input e output

Ma i comandi non vivono nel vuoto.

Problema concreto:
- un comando funziona in una shell ma non in un’altra
- uno script si comporta diversamente su due macchine
- un eseguibile “non viene trovato”
- una configurazione sembra sparire

Contesto reale:
- scripting
- ambienti di sviluppo
- server
- automazione

Vincoli noti:
- la shell mantiene uno stato
- ogni processo eredita un contesto
- non tutto è hardcoded nei comandi

Cosa **non** sappiamo ancora fare:
- capire da dove arrivano certi valori
- controllare il contesto di esecuzione
- rendere prevedibile il comportamento dei comandi

Perché il problema è rilevante:
> Senza comprendere variabili e ambiente, la CLI è fragile e non riproducibile.

---

## Teoria

Spiegazione dei concetti necessari **solo** per controllare il contesto di esecuzione.

---

### Variabili di shell

Una **variabile di shell** è una coppia chiave/valore mantenuta dalla shell.

Esempio concettuale:
```

NOME=valore

```

Caratteristiche:
- esiste solo nella shell corrente
- non è automaticamente visibile ai processi figli
- influenza il comportamento dei comandi

---

### Assegnazione e lettura

Assegnazione:
```

VAR=ciao

```

Lettura:
```

$VAR

```

La shell **espande** le variabili prima di eseguire il comando.

---

### Variabili d’ambiente

Una **variabile d’ambiente** è una variabile:
- esportata dalla shell
- ereditata dai processi figli

Differenza chiave:
> tutte le variabili d’ambiente sono variabili di shell,  
> ma non tutte le variabili di shell sono variabili d’ambiente.

---

### Export

Per rendere una variabile disponibile ai processi figli:

```

export VAR=valore

````

Da questo momento:
- ogni comando lanciato dalla shell la vede
- gli script possono usarla

---

### Ambiente di esecuzione

Ogni processo ha un **ambiente**, composto da:
- variabili
- directory corrente
- permessi
- stdin/stdout/stderr

Questo ambiente viene:
- ereditato
- eventualmente modificato
- poi distrutto alla fine del processo

---

### PATH

`PATH` è una variabile d’ambiente fondamentale.

Contiene:
- una lista di directory
- separate da `:`

Quando esegui un comando:
1. la shell legge `PATH`
2. cerca l’eseguibile in ordine
3. esegue il primo trovato

Se non è in `PATH`:
- il comando **non esiste** per la shell

---

## Esempi

Applicazione pratica immediata dei concetti teorici.

---

### Definire e leggere una variabile

```bash
MSG=hello
echo $MSG
````

---

### Variabile non esportata

```bash
VAR=test
bash
echo $VAR
```

Output:

```
(empty)
```

La variabile non è stata esportata.

---

### Variabile esportata

```bash
export VAR=test
bash
echo $VAR
```

Ora è visibile.

---

### Visualizzare l’ambiente

```bash
env
```

Oppure:

```bash
printenv
```

---

### Ispezionare PATH

```bash
echo $PATH
```

Verificare dove si trova un comando:

```bash
which ls
```

---

### Eseguire un comando con ambiente temporaneo

```bash
VAR=demo command
```

La variabile vale **solo per quel comando**.

---

### Comando non trovato

```bash
mytool
```

Errore tipico:

```
command not found
```

Soluzioni:

* percorso assoluto
* aggiungere la directory a `PATH`
* eseguire con `./`

---

## Output atteso della lezione

Alla fine di questa lezione sai:

* distinguere variabili di shell e d’ambiente
* controllare il contesto di esecuzione
* capire perché un comando “non esiste”
* usare PATH in modo consapevole
* rendere script e comandi riproducibili

Da qui in poi, la CLI diventa
**controllata e prevedibile**.
