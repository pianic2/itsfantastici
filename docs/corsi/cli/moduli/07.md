# Lezione 07 — Processi e Job Control

## Problema

I comandi ora funzionano, le pipeline anche, l’ambiente è sotto controllo.  
Ma quando lanciamo un comando, **cosa succede davvero**?

Problema concreto:
- un comando “rimane bloccato”
- un processo consuma troppe risorse
- vogliamo eseguire più comandi in parallelo
- dobbiamo fermare o riprendere un’esecuzione

Contesto reale:
- server
- script lunghi
- analisi di log
- programmi interattivi

Vincoli noti:
- ogni comando avvia un processo
- il sistema esegue più processi contemporaneamente
- non tutto gira in primo piano

Cosa **non** sappiamo ancora fare:
- capire quali processi sono in esecuzione
- controllare processi bloccati o lenti
- gestire esecuzioni in background

Perché il problema è rilevante:
> Senza controllo dei processi, la CLI diventa opaca e ingestibile.

---

## Teoria

Spiegazione dei concetti necessari **solo** per capire e controllare l’esecuzione dei comandi.

---

### Cos’è un processo

Un **processo** è:
- un programma in esecuzione
- con un proprio stato
- un proprio identificatore (**PID**)
- un proprio ambiente

Ogni comando lanciato dalla shell crea **almeno un processo**.

---

### Ciclo di vita di un processo

Un processo:
1. viene creato
2. viene eseguito
3. termina

Alla fine restituisce un **exit code**:
- `0` → successo
- diverso da `0` → errore

La shell usa l’exit code per decidere cosa fare dopo.

---

### Foreground e background

- **Foreground**: il processo occupa il terminale
- **Background**: il processo gira senza bloccare la shell

La shell gestisce questa distinzione, non il sistema operativo.

---

### Job

Un **job** è:
- uno o più processi
- gestiti dalla shell
- associati a un terminale

Il job control è una funzionalità **della shell**, non dei processi.

---

### Segnali

I **segnali** sono messaggi inviati ai processi.

Alcuni segnali comuni:
- `SIGINT` → interruzione (`Ctrl+C`)
- `SIGTERM` → terminazione gentile
- `SIGKILL` → terminazione forzata
- `SIGSTOP` / `SIGCONT` → pausa / ripresa

I segnali servono a **controllare**, non a distruggere a caso.

---

## Esempi

Applicazione pratica immediata dei concetti teorici.

---

### Visualizzare i processi con `ps`

Per capire cosa sta girando sul sistema possiamo usare comandi che forniscono o un’istantanea dello stato dei processi, o una vista dinamica e aggiornata. Usate questi strumenti come un’iniezione di realtà: osservate UID, PID, genitore, uso CPU/MEM e la riga di comando del processo per decidere l’azione successiva.

Esempi:
```bash
ps
```

```bash
ps -ef
```

```bash
ps aux

USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1 169004  8128 ?        Ss   09:00   0:01 /sbin/init
pianic2   2345  0.1  0.5 276120 22044 pts/0    S+   10:12   0:00 bash
pianic2   2389  2.3  1.2 512000 51200 pts/0    Sl   10:15   0:05 /usr/bin/python3 myscript.py --flag
```

```bash
top
```

Premi `q` per uscire da `top`.

### Avviare un processo in background

```bash
sleep 60 &
```

La shell torna subito disponibile.

---

### Elenco dei job

```bash
jobs
```

Output tipico:

```
[1]  Running  sleep 60 &
```

---

### Portare un job in foreground

```bash
fg %1
```

---

### Mettere in background un processo

Durante l’esecuzione:

```text
Ctrl+Z
```

Poi:

```bash
bg
```

---

### Terminare un processo

Con PID:

```bash
kill 1234
```

Forzato:

```bash
kill -9 1234
```

Regola pratica:

> usa `-9` solo se il processo non risponde.

---

### Terminare per nome

```bash
killall sleep
```

---

### Verificare l’exit code

```bash
false
echo $?
```

Output:

```
1
```

L’exit code è parte del contratto tra processi.

* distinguere foreground e background
* usare job control
* inviare segnali in modo corretto
* interpretare exit code

Ora la CLI diventa
**interattiva, concorrente e sotto controllo**.
